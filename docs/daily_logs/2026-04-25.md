#### 2. PnL Tracker の重大バグ修正
**問題発見**:
- 確定損益 $185,694 vs Broker 実績 $1,885（**99倍の乖離**）

**原因特定**:
- 46件の取引で `entry_price = 0` が記録
- `paper_demo.py` で quote 取得失敗時に 0.0 を設定
- $181,401 の誤った利益を計上

**修正実施**:
- 無効な取引（entry_price = 0）を除外
- State を再構築・バックアップ保存

**修正結果**:
- 確定損益: $185,694.48 → $4,292.74
- 決済済み取引: 90件 → 44件
- 勝率: 82.22% → 63.64%

#### 3. entry_price=0 の根本原因修正と再発防止
**根本原因**:
- `paper_demo.py`: quote 取得失敗時に `entry_price = 0.0` を `record_submission` に渡していた

**再発防止策**:
1. `paper_demo.py`: entry_price <= 0 なら record_submission をスキップ、警告ログ出力
2. `pnl_tracker.py`: `record_submission` で price <= 0 なら ValueError を raise
3. テスト実施: price=0 で ValueError、price>0 で正常動作を確認

**効果**: entry_price=0 の取引は二度と作成されない

#### 4. 残り $2,407 差異の原因分析
**差異**:
- Broker 実績: $1,885.35（真実）
- Tracker: $4,292.74
- 差異: $2,407.39（2.3倍 over-estimation）

**原因判明**:
1. **複数エントリー/単一エグジット**: MRVL 15回など、同じ symbol を複数回売買
2. **Entry price 計算差**: bid/ask 中値 vs 実約定価格
3. **手数料未考慮**: tracker が手数料を計算していない

**対応方針**:
- 短期: Broker equity を信頼、tracker は参考値として使用
- Week 2: T5 で partial fill + FIFO/LIFO 実装で正確な PnL 計算

### T7. Low Conversion 分析・改善
**分析結果**:
- **問題 1**: Position size limit $50 が厳しすぎる
  - 平均リクエスト: $113
  - 全 rejection の 100% がこの理由
  - 96件の decisions が却下

- **問題 2**: 既存ポジション保有による重複回避
  - PLTR, PATH, CRM, DDOG など保有中は追加買い禁止
  - 強い追加シグナル (confidence 0.85, strength 1.00) も無視

**実施した改善**:
1. **max_position_size: $50 → $400** (8倍)
   - Risk rejection: 96 → 0 (100% 削減)
   - Conversion rate: 13.2% → 16.2% (+3.0%)

2. **Symbol-level limit: 10% per symbol**
   - 既存ポジションへの追加買い 78件が可能に
   - PLTR (10.0%) は BLOCKED、CRM (7.3%), ARM (1.3%) は ALLOWED
   - 過度な集中を防止しつつ pyramiding を許可

**期待される総合効果**:
- Conversion rate: 13.2% → 20-25% (目標 30% に近づく)
- Submissions: 66 → 100-120 orders/period
- リスク管理: 維持 (各 symbol 10% 上限)
